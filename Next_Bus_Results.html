User Task Analysis

Use Cases

To outline the user's process as they navigate through the interface, we wrote use cases for the following tasks:

  1. Find the next available bus to take the user home (via multiple bus agencies)
  2. Create a user profile (to indicate home, favorites, and frequent locations)
  3. Open help menu for navigation

We also created hierarchical task analyses for each use case. For brevity, I will only outline one use case.

Use Case 1

A: The user opens NextBus.

  1. The user opens the application.
  2. The user finds the app on their phone.
  3. The user selects the app.

B: The user plans route.

  1. The user finds transit options.
  2. The user selects Transit menu option.
  3. The user selects the agency (one or more than one).
  4. The user selects the route.
  5. The user specifies the destination.
  6. The user specifies the stop.
  7. The user finds nearby options.
  8. The user selects the nearby menu option.
  9. The user filters through nearby menu options in the direction of their destination.
  10. The user selects the ideal route.
  11. The user finds the fastest route.

C: The user travels to bus stop

  1. The user revisits step B1 if the route does not work for the user’s schedule.
  2. The user locates the bus stop.
  3. The user opens the map.

Design Recommendations

To improve the UI in the NextBus app, we identified four design recommendations:

Recommendation 1: Combine and reduce list view and map view

Currently, the “Nearby” tab within the app serves as the primary navigation page. This page contains a list view displaying all nearby routes, sorted by proximity from the user’s location. When clicking into a route, the user is shown a map view with the selected route overlayed and highlighted in color. The route details are displayed above the map view. These details are identical to the details listed in the Nearby list view.

Figure 1: Current NextBus Page

To increase readability and efficiency at first glance, we propose 1) combining these views into one view which then allows us to 2) remove route details that is now shared by one view.

The primary motive for this design decision was efficiency. Combining the map and list view minimizes the number of discrete steps necessary to compare routes, maximizing exposure to route information, and ultimately reduces the amount of time required for the user to identify the optimal route. The second motive in collapsing the two screens into one is that it reduces extraneous information such as the route detail that was previously duplicated on the list view and map view.

Figure 2: Home Page Redesign

Recommendation 2: Improve icons, icon labels, and placement

We wanted to increase usability for icons by helping users to understand the icons’ controls and impacts on the user interface. Also, we want to ensure that the selected icons are not confusing to the user. For example, the Nearby icon represents a dropped pin on a map. However, the dropped pin concept is unique to digital maps. Although this icon is frequently used in transit apps, first time users may not understand the significance of this app.

Figure 3: Old Icons

Thus, we must change the icon so that users understand how the icon controls the interface. To accomplish this goal, we plan to change the icons from symbolic style to iconic style.

Figure 4: New Icons

Recommendation 3: Select Multiple Agencies At Once

This design decision enables users to select multiple bus agencies either during the onboarding process or at any time in their profile page. Currently the list view of agencies allows user to tap on a single agency, which applies a checkmark icon to indicate the preference - effectively serving as a radiobutton.

Instead of permitting users to view bus routes from a single bus agency in the primary navigation page, we allow them toggle the bus agency through a dropdown list at the top right corner of the navigation page. This dropdown will contain all the agencies they previously selected, and the map and route list will be serve data from the selected agency only.

Figure 5: Selecting Multiple Agencies at Once

Recommendation 4: Increase Customization

In “Customization of user interfaces to reduce errors and enhance user acceptance,” (Burkoleter, Weyers, Kluge, and Luther, 2014) the authors test the impact of customization on users’ efficiency, visual element recognition, and error rates. In their experiment, they conducted a between subjects experiment between two user groups: a customizable interface and a standard one. The participants in the customizable group rearranged their own interfaces to their preference. Overall, the researchers found that the customizable group exhibited a lower error rate and higher acceptance rate in relation to their interface.

Thus, we used this study to create a prototype profile option that enables users to achieve customization.

Contextual Inquiry

After creating the prototype, we conducted a small contextual inquiry to survey users about their impressions of NextBus. Out of the users we surveyed, three of them were active NextBus users and one was new to the platform. Each user met face-to-face with a team member, and were shown the existing NextBus mobile application.

First, we asked them about their general impressions of the application. Then, we asked the users about issues and desired changes to existing functionality. Next we asked users to complete the following tasks:

  • Find specific route and accompanying map
  • Edit application settings
  • Set a frequent destination

Afterwards, we showed the users our prototype and asked them about their impressions. Since this is a small study, we acknowledge that we cannot draw any generalizations from this test. From the contextual inquiry, we learned the following insights: Strengths of the redesign include the simplification of navigation flow and the comprehensiveness of user profile

Weaknesses included the order of menu buttons and the lack of support features for new users. For individuals who were active NextBus users, their existing preferences and habits affected their statements. As an example, combining the map view and list view to form the primary navigation flow was highly supported by the comments of some active users, while others did not find any improvement as they did not use the map at all.

Conclusion

From this experience, I believe the changes that we implemented will improve usability for the user. The most important lesson I learned from this experience is the value of cognitive load. When users navigate a user interface, they only need to know the see the essential information. Thus, cognitive load can make or break a user’s experience. If a user finds it mentally taxing to use an app, then they will not use the app.
